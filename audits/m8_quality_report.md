### Methodology

- **compiles-correct (92):** 88 hand-written files ≥50 lines with no cleanup tags, plus 4 smaller clean hand-written files. These have full FSM logic, channel declarations, register state, and control flow.
- **compiles-partial (141):** 131 auto-generated files ≥15 lines that contain at least one logic statement (`send`, `match`, `if`), plus 10 semi-manual conversions. These have structural scaffolding and some logic but typically use simplified/approximate patterns.

- **needs-cleanup (50):** All hand-written; contain `MANUAL-CLEANUP-NEEDED` comments marking spots where the conversion was incomplete or uncertain. These range from nearly-correct files with one cleanup tag to files with many unresolved sections.

## Detailed Audit of 5 Representative Files

### 1. SyncSpRam.anvil (RAM/Memory)

- **Origin:** Hand-written
- **Lines:** 57
- **Compiles:** Yes (with 4 warnings about non-constant array offsets)
- **Rating: Correct**
- **Analysis:**
  - Declares a 1024×32 single-port RAM (`reg mem : sp_word[1024]`)
  - Implements chip-select, write-enable, address, write-data, read-data channels
  - Read/write logic correctly guarded by `csel` and `wren` — matches SV semantics
  - Uses the recv-then-register pattern with explicit cycle boundaries
  - **Issue:** Sequential recv per channel adds latency vs. SV's parallel signal sampling. Array offset warnings are non-fatal.

### 2. axi_adapter.anvil (AXI/Bus)

- **Origin:** Hand-written
- **Lines:** 252
- **Compiles:** Yes
- **Rating: Partial (70% logic coverage)**
- **Analysis:**
  - Implements 10-state FSM (IDLE through WAIT_AMO_R_VALID) — matches SV
  - Write path FSM transitions are complete (IDLE → WAIT_B_VALID / WAIT_AW_READY / etc.)
  - Read path with burst counter and cache line assembly (4×64→256) is present
  - **Issues:**
    - `send out.rdata (*r_wdata)` on line 160 sends write data instead of assembled cache line — should be concatenation of `cache_line0..3`
    - Burst counter `cnt` is never decremented in the FSM — missing `set cnt := *cnt - 2'd1` on W_READY
    - AMO state (9) transitions back to IDLE without computing AMO result
    - No address offset computation for read alignment

### 3. multiplier.anvil (Arithmetic)

- **Origin:** Hand-written
- **Lines:** 124
- **Compiles:** Yes
- **Rating: Partial (60% logic coverage)**
- **Analysis:**
  - Pipeline structure correct: input staging → combinational compute → pipeline register → output mux
  - Sign extension logic present (sign_a, sign_b based on MULH/MULHSU)
  - **Issues:**
    - Multiplication is unsigned-only (`logic[64] * logic[64]`) — sign extension bits not actually prepended to operands. SV original does `signed'({a[31]&sign_a, a}) * signed'({b[31]&sign_b, b})`
    - Output mux for MULH/MULHU/MULHSU all extract the same bits — should extract upper 32 bits for MULH* variants vs lower 32 for MUL
    - MULW (32-bit on RV64) not handled separately
    - CLMUL hardwired to zero (correct, since RVB=0)

### 4. load_unit.anvil (Control/FSM)

- **Origin:** Hand-written
- **Lines:** 453
- **Compiles:** Yes
- **Rating: Partial (75% logic coverage)**
- **Analysis:**
  - Full 9-state FSM with all transitions (IDLE, WAIT_GNT, SEND_TAG, WAIT_PAGE_OFFSET, ABORT_TRANSACTION, ABORT_TRANSACTION_NI, WAIT_TRANSLATION, WAIT_FLUSH, WAIT_WB_EMPTY)
  - Load buffer with 2 entries — allocation, freeing, flushing all implemented
  - Translation request, data request, kill request, tag valid outputs correctly driven per state
  - **Issues:**
    - Sign extension for load results is missing — `let result = shifted` with no LB/LBU/LH/LHU/LW/LWU muxing
    - Data shift logic (`>> shift_amt`) approximates byte alignment but doesn't mask unused bits
    - Sequential recv of 20+ channels in a single loop adds significant latency vs. SV's parallel always_comb

### 5. riscv_pkg.anvil / ariane_pkg.anvil (Package/Type Definitions)

- **Origin:** Auto-generated (`sv2anvil.py`)
- **Lines:** 12 each
- **Compiles:** Yes
- **Rating: Stub**
- **Analysis:**
  - Both are empty shells: `proc riscv_pkg() { loop { cycle 1 } }`
  - The original SV packages define hundreds of constants, enums, typedefs, and structs (instruction opcodes, CSR addresses, exception causes, pipeline types)
  - None of this content was converted — these are pure stubs
  - **Contrast:** `axi_pkg.anvil` (semi-manual, 120 lines) correctly defines type aliases, enums, and structs. The package conversion problem is solvable but wasn't applied to `riscv_pkg` or `ariane_pkg`.

## Cross-Cutting Issues

These issues were also identified by Iris's prior audit and confirmed here:

1. **Channel-per-signal anti-pattern:** Every SV port becomes a separate channel with individual `recv`/`send`. This adds latency (sequential recv) and port bloat vs. using structs or bundled channels.

2. **`@dyn-@dyn` sync annotations:** All channels use dynamic-to-dynamic timing. Pipeline stages should use `@#1-@#1` for proper clock-cycle semantics.

3. **Sequential recv adds latency:** A module with 20 input channels does 20 sequential `recv` operations per cycle. In SV, all inputs are sampled simultaneously.

4. **Package stubs:** 149 auto-generated files are empty stubs (<15 lines). This includes all config packages, most UVM packages, and critical type definition files (`riscv_pkg`, `ariane_pkg`). Without these, dependent modules cannot reference shared types.

5. **Missing sign extension:** Arithmetic modules (multiplier) and load paths (load_unit) skip sign extension, which is critical for RISC-V correctness.

6. **MANUAL-CLEANUP-NEEDED prevalence:** 50 files (all hand-written) contain these tags. The hand-written files represent the best-quality conversions, yet 35% of them still have known incomplete sections.

## Recommendations

1. **Priority: Convert package stubs.** `riscv_pkg` and `ariane_pkg` are foundational — nearly every module references types from these packages. The `axi_pkg.anvil` semi-manual conversion shows this is feasible.

2. **Resolve MANUAL-CLEANUP-NEEDED tags.** The 50 flagged files are the highest-quality conversions and closest to correct. Fixing their cleanup items would move them from "needs-cleanup" to "correct."

3. **Fix sign extension in arithmetic paths.** The multiplier and load unit both skip sign extension, which will produce incorrect results for signed operations.

4. **Address channel-per-signal pattern.** Bundle related signals into structs/records to reduce latency and port count. This is a structural issue affecting all hand-written files.

5. **Replace `@dyn-@dyn` with explicit timing.** Pipeline stages need `@#1-@#1` for correct clock-domain semantics.
